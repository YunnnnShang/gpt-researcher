# 云产品深度分析框架

根据用户提供的**云厂商 + 产品名称**（如 "AWS Bedrock"），进行全面的产品分析。

## 支持的云厂商

| 厂商 | 英文名称 | 中文简称 | 官方域名 |
|------|----------|----------|----------|
| Amazon Web Services | AWS | 亚马逊云服务 | aws.amazon.com, docs.aws.amazon.com |
| Microsoft Azure | Azure | 微软云 | azure.microsoft.com, learn.microsoft.com |
| Google Cloud | GCP | 谷歌云 | cloud.google.com |
| 阿里云 | Alibaba Cloud | 阿里云 | aliyun.com |
| 华为云 | Huawei Cloud | 华为云 | huaweicloud.com |
| 腾讯云 | Tencent Cloud | 腾讯云 | cloud.tencent.com |
| 火山引擎 | Volcengine Cloud | 火山引擎 | www.volcengine.com |

**重要**: 本框架**优先使用模型自带 WebSearch 工具**进行信息收集，如调用失败则回退到博查搜索 API。

## 执行前置要求

1. **确认实际日期**：使用 `date +"%Y-%m-%d"` 命令获取当前实际日期，**报告中所有日期必须基于实际日期**
2. **识别云厂商**：根据用户提供的产品名称，识别所属云厂商（可通过产品名称特征判断）
3. **进度追踪**：使用 TodoWrite 工具创建任务列表，并随进度实时更新状态
4. **信息收集**：**使用 WebSearch 工具**

   单次查询
   ```
   WebSearch: "厂商产品名 关键词 site:官方域名"
   ```

   批量搜索时，可在一个消息中并行调用多个 WebSearch：
   ```
   WebSearch: "AWS Bedrock features site:aws.amazon.com"
   WebSearch: "AWS Bedrock announces site:aws.amazon.com/blogs/"
   WebSearch: "AWS Bedrock re:Invent 2025"
   ```

5. **最终输出**：完成后将报告保存为 Markdown 文档，文件名包含实际日期

## 分析维度

### 一、产品定位与核心价值
1. **定位官网产品相关页面**
   - **产品页面**: `aws.amazon.com/产品名/` 
   - **特性页面**: `aws.amazon.com/产品名/features/`
   - **FAQ页面**: `aws.amazon.com/产品名/faqs/` 
   - **新特性发布页面**: `https://aws.amazon.com/new/` ，且页面中有Search选项。
   - **Press release新闻稿页面**: `press.aboutamazon.com/aws?q=产品名` 
   - **Blog页面**: `aws.amazon.com/blogs/` ，且页面中有Search选项。

2. **核心价值主张**
   - 产品解决的**核心问题**是什么？
   - **目标客户群体**（企业规模、行业类型、技术角色）
   - 与同类产品的**差异化定位**

### 二、产品关键特性（Top 10-15）- 深度细节提取
1. **特性信息收集**
   - 查看厂商 "特性页面"/"FAQ页面"
   - **搜索策略（优先使用 WebSearch）**：
     * 查询：`"产品名 features capabilities site:aws.amazon.com"`
     * 查询：`"产品名 announces launches site:aws.amazon.com/blogs/"`
     * 如 WebSearch 失败，使用博查 API：`include: "aws.amazon.com|docs.aws.amazon.com"`
   - **信息源限制**: 仅使用第一方官方来源
     - ✅ 官方网站 (aws.amazon.com, azure.microsoft.com 等)
     - ✅ 官方博客 (aws.amazon.com/blogs/, cloud.google.com/blog/ 等)
     - ✅ 官方文档 (docs.aws.amazon.com, learn.microsoft.com 等)
     - ✅ 高质量第三方媒体 (如InfoQ, 腾讯新闻可以使用)
     - ❌ 一般第三方媒体等均不使用


2. **特性细节提取要求**
   每个特性必须包含以下**细节信息**：

   | 维度 | 提取内容 | 来源示例 |
   |------|----------|----------|
   | **功能名称** | 特性的官方名称 | 新闻稿标题、产品页面 |
   | **发布时间** | 具体发布日期 | "2025年12月11日发布" |
   | **核心能力** | 详细描述功能做什么 | "支持X模型，提供Y能力" |
   | **技术细节** | 实现方式、技术规格 | "基于X技术，支持Y参数" |
   | **量化指标** | 性能数字、对比数据 | "提升85%准确率"、"延迟降低50%" |
   | **使用场景** | 具体应用示例 | "适用于X场景，如Y案例" |
   | **客户价值** | 业务价值、ROI | "降低成本X%、提升效率Y倍" |
   | **引用来源** | 具体的 URL | [官方博客链接](URL) |

3. **特性筛选与排序**
   - 按**客户价值**和**影响力**维度排序（而非技术实现）
   - 优先选择：最新发布的、有量化数据的、有客户案例的特性
   - 筛选出10-15个**最受关注**的特性

4. **特性描述模板**（用于报告）
   ```markdown
   ### [特性官方名称]
   - **发布时间**: YYYY-MM-DD
   - **核心能力**: [详细描述，2-3句话]
   - **技术细节**: [实现方式、关键参数]
   - **性能指标**: [量化数据，如"提升X%"]
   - **适用场景**: [具体应用场景]
   - **客户价值**: [业务价值、ROI]
   - **来源**: [官方新闻稿/博客标题](URL)
   ```

### 三、最近一年产品动态
1. **新特性发布**
   - 查看厂商 "新特性发布页面" 
   - 搜索产品相关的新功能发布，聚焦最近12-18个月（需确认当前的实际日期）

2. **官方博客与公告**
   - 查看厂商 "Blog页面、公众号"
   - 提取产品更新、技术演进、客户案例信息

### 四、客户场景与案例
1. **典型应用场景**
   - 官方文档中的使用场景
   - 行业最佳实践

2. **客户案例**
   - 官方发布的成功案例
   - 案例中的业务挑战与解决方案

## 工作原则
1. **优先使用 WebSearch**: 首先使用模型自带 WebSearch 工具，使用 `site:` 语法限定官方域名
2. **后备使用博查 API**: 如 WebSearch 失败，使用博查 API 的 `include` 参数限定搜索范围
3. **仅使用第一方官方来源**: 所有搜索结果必须来自官方域名
4. **时效性**: 优先使用近一年的信息，确保包含最新年度大会/重要发布
5. **客观性**: 基于官方发布的事实描述，避免主观臆断
6. **可操作性**: 提供具体的官方链接，方便用户深入了解

## 常见陷阱提醒

### 0. 搜索工具优先级（最重要）
- **第一优先**: 使用 WebSearch 工具，配合 `site:` 语法限定官方域名，如 `"AWS Bedrock features site:aws.amazon.com"`
- **第二优先**: 如 WebSearch 失败，使用博查 API 的 `include` 参数限定搜索范围
- **示例**: WebSearch 调用
  ```
  WebSearch: "AWS Bedrock features site:aws.amazon.com"
  ```

### 1. 日期错误
- 必须：使用 `date +"%Y-%m-%d"` 确认实际日期
- 避免：凭记忆或假设日期

### 2. 遗漏最新事件
- 年度大会要搜索最新已发生的年度
- 如果当前是 2026-01，AWS 应搜索 re:Invent 2025（非 2024）
- 不同厂商大会时间不同，参考上表

### 3. 信息交叉验证
- 关键数据（性能提升、成本降低）需多源验证
- 发布日期以官方页面为准，非转载文章
- 技术细节以官方文档为准，非新闻稿简化描述

### 4. 报告一致性
- 报告中的所有日期应与实际日期逻辑一致
- 不能出现未来日期描述为"最近"

## 从新闻稿/Blog提取细节的操作指南

### 步骤 1: 识别信息源类型
- **官方新闻稿**: 通常包含" announces"、" launches"、" now available"、"正式发布"等关键词
- **官方博客**:
  - AWS: `aws.amazon.com/blogs/`
  - Azure: `azure.microsoft.com/blog/`
  - GCP: `cloud.google.com/blog/`
  - 阿里云: `yq.aliyun.com/`
  - 华为云: `huaweicloud.com/about/news/`
  - 火山引擎: `www.volcengine.com/news/`  
- **产品发布页面**:
  - AWS: `aws.amazon.com/about-aws/whats-new/`
  - Azure: `azure.microsoft.com/updates/`
  - GCP: `cloud.google.com/blog/products/`
  - 阿里云: `aliyun.com/new/`
  - 华为云: `huaweicloud.com/product/`
  - 火山引擎: `www.volcengine.com/product/`

### 步骤 2: 提取关键信息要素

**从新闻稿标题提取**:
- 产品名称 + 动词 (launches, announces, releases, 正式发布)
- 新功能名称
- 目标用户/场景

**从正文首段提取**:
- 问题背景：解决什么痛点
- 解决方案概述：新功能的核心能力
- 关键指标：量化数据（百分比、倍数、数量）

**从引用语录提取**:
- 产品经理/高管的引言通常包含：
  * "我们现在可以..." → 客户价值
  * "相比之前提升X%" → 性能指标
  * "客户告诉我们..." → 市场需求

**从技术细节部分提取**:
- 实现方式、技术架构
- API 变更、配置参数
- 兼容性要求、限制条件

### 步骤 3: 验证和交叉引用
- 关键数据（性能提升、成本降低）需多源验证
- 发布日期以官方页面为准，非转载文章
- 技术细节以官方文档为准，非新闻稿简化描述

### 步骤 4: 记录来源
- 每个特性必须记录来源 URL
- 优先引用官方页面，其次是权威媒体报道
- 格式：`[页面/文章标题](URL)`
