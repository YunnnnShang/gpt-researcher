## 执行前置要求

1. **确认实际日期**：使用 `date +"%Y-%m-%d"` 命令获取当前实际日期，**报告中所有日期必须基于实际日期**
2. **识别云厂商与产品**：根据用户提供的信息，识别所属云厂商和产品线
3. **进度追踪**：使用 TodoWrite 工具创建 6 个分析维度任务，并随进度实时更新状态
4. **信息收集**：**优先使用 WebSearch 工具**，使用 `site:` 语法限定官方域名

   **方式一：WebSearch 工具（推荐优先使用）**
   ```
   # 特性基础信息
   WebSearch: "厂商 产品 特性名称 features site:官方域名"

   # 技术文档
   WebSearch: "厂商 产品 特性名称 documentation site:文档域名"

   # 客户案例
   WebSearch: "厂商 产品 特性名称 customer case study site:官方域名"

   # 竞争对比
   WebSearch: "厂商 产品 特性名称 vs 竞品 comparison"
   ```

   **批量搜索示例**：
   ```
   WebSearch: "AWS S3 Intelligent-Tiering features site:aws.amazon.com"
   WebSearch: "AWS S3 Intelligent-Tiering pricing site:aws.amazon.com"
   WebSearch: "AWS S3 Intelligent-Tiering customer case study site:aws.amazon.com"
   WebSearch: "AWS S3 Intelligent-Tiering documentation site:docs.aws.amazon.com"
   ```

5. **最终输出**：完成后将报告以 Markdown 文档形式输出

## 分析维度

### 维度 1：特性归属与分类定位

**目标**：确定该特性在产品特性树中的位置，理解其分类逻辑

**分析要点**：
1. **产品特性树定位**
   - 该特性属于哪个功能模块？（如：存储类 > 灾备）
   - 特性层级：基础特性（免费）vs 增值特性（付费）

2. **版本与可用性**
   - 发布日期与版本历史
   - 地域可用性（哪些 Region 支持）
   - 服务等级承诺（SLA）

3. **依赖关系**
   - 前置依赖：需要开启哪些其他特性或配置？
   - 互斥关系：与哪些特性不能同时使用？
   - 关联特性：通常与哪些特性组合使用？

**信息来源**：
- 产品文档的 "Features" 或 "What's New" 页面
- 产品架构白皮书
- 官方博客的发布说明

---

### 维度 2：场景问题与解决方案

**目标**：深入理解该特性解决的具体业务场景和技术问题

**分析要点**：
1. **核心痛点识别**
   - 在没有该特性之前，用户面临什么问题？
   - 问题的频率、影响范围和成本
   - 传统解决方案的局限性

2. **目标用户画像**
   - 主要用户角色（开发者/运维/架构师/业务人员）
   - 用户技术成熟度要求
   - 典型用户规模（初创/中型/大型企业）

3. **适用场景矩阵**

   | 场景类型 | 具体描述 | 适用度 |
   |----------|----------|--------|
   | 场景1 | [描述] | ⭐⭐⭐⭐⭐ |
   | 场景2 | [描述] | ⭐⭐⭐⭐ |
   | 场景3 | [描述] | ⭐⭐⭐ |

4. **不适用场景**
   - 明确说明哪些场景不适合使用该特性
   - 可能的风险或限制

**信息来源**：
- 官方文档的 "Use Cases" 或 "Best Practices"
- 产品白皮书中的场景描述
- 技术博客中的案例分析

### 维度 3：客户案例与价值验证

**目标**：收集真实的客户应用案例，量化业务价值

**分析要点**：
1. **官方客户案例**
   - 客户名称（可公开的部分）
   - 所属行业
   - 业务挑战
   - 解决方案（如何使用该特性）
   - 量化收益（成本节省、效率提升、性能改善等）

2. **价值主张量化**

   | 价值维度 | 具体指标 | 典型案例数据 |
   |----------|----------|--------------|
   | 成本节省 | 存储成本降低 X% | 案例A：节省 40% |
   | 效率提升 | 操作时间减少 Y% | 案例B：节省 60% 运维时间 |
   | 性能改善 | 延迟降低 Z ms | 案例C：降低 30% |
   | 可靠性 | 可用性提升 | 案例D：达到 99.99% |


**信息来源**：
- 官方 "Customer Stories" 或 "Case Studies"
- 合作伙伴发布的联合案例
- 技术大会（re:Invent, Build, 云栖大会等）的演讲

## 工作原则

1. **优先使用 WebSearch**: 首先使用模型自带 WebSearch 工具，使用 `site:` 语法限定官方域名
2. **后备使用博查 API**: 如 WebSearch 失败，使用博查 API 作为后备
3. **仅使用第一方官方来源**: 所有技术细节必须来自官方文档，避免使用第三方技术博客
4. **区分事实与观点**: 官方文档中的内容为"事实"，竞争对比中的评价需标注为"分析"
5. **时效性**: 优先使用近一年的信息，技术规格以最新官方文档为准
6. **完整性**: 6个分析维度缺一不可，如某维度信息缺失需明确标注"未找到官方信息"

## 常见陷阱提醒

### 1. 特性名称混淆
- 同一特性在不同文档中可能有不同名称（如英文/中文/缩写）
- 需要确认特性的唯一标识符

### 2. 版本差异
- 确认分析的是最新版本特性
- 注意不同地域的版本差异

### 3. 免费 vs 付费
- 明确区分免费功能和付费功能
- 注意隐藏的按量计费项

### 4. 预览版 vs GA
- 注意区分 Preview/Beta 和 GA (General Availability)
- 预览版特性可能有功能限制或随时变更

### 5. 地域差异
- 某些特性只在特定 Region 可用
- 价格和功能在不同地域可能有差异

### 6. 竞争对比的客观性
- 避免过度偏向本厂商
- 竞品信息需从竞品官方文档确认
- 使用"据官方文档"等限定语
